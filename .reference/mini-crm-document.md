# ミニCRM

## UI/UX Case Study（プロトタイプ制作）

**案件（スポンサー）→請求→入金を“漏れなく回す”ための小さなCRM**

* 種別：B2B / 管理画面
* 担当：業務整理 / UX設計 / 実装（予定）

---

## 背景：スプシ運用は「回るほど事故る」

* 情報が **スプシ / Slack / カレンダー / Drive** に散り、**最新が分からない**
* 「担当は誰？次は何？」がSlack往復になり、**判断が遅れる**
* 月末の棚卸しで **請求漏れ・確認漏れ**が発覚しがち

> 回るほど、人の注意力に依存して“漏れ”が増える

---

## 気づき：必要なのは「台帳」ではなく“運用のOS”

* スプシは自由度が高い一方で、
  **入力ゆれ / 更新漏れ / 重複 / 数式破損 / 担当不明**が起きやすい
* 事故を減らすには、
  **状態（ステータス）・必須条件・一意性・履歴・復旧導線**を仕組み化する必要がある

> ルールを画面に埋め込むと、現場の判断コストが下がる

---

## 制作プロセス：プロトタイプで“触って固める”

アイデア段階でも前に進むために、まず **主要フローが一周できるプロトタイプ**を作る。

1. **目的と課題を1枚に整理**（誰の何を変えるか）
2. **体験の核（主要フロー）を固定**（迷わない道筋）
3. **Figmaでクリック可能プロトタイプ**（一周できるまで）
4. **実装→Preview→レビュー→調整**（小さく出して学ぶ）
5. **例外・エラーの回復導線を整備**（失敗しても止まらない）

* **スプリントごとに「触れるもの」または「意思決定できる資料」を必ず出す**

*注：受託でも同様に、まず「主要フロー一周のプロト」から着手できます（アイデア段階から相談OK）。*

---

## PHASE1：要件定義とプロトタイプ

プロトタイプを「仕様の代わり」にして、言葉の曖昧さを減らす。

* **まず決める**：誰が / 何を / いつまでに / どう回すか
* **まず作る**：主要フローが“一周”できるクリックプロト
* **まず直す**：触って分かった **迷い / 漏れ / 例外** を潰す

> 仕様書より先に「触って合意」して、手戻りを減らす

## PHASE1：レビューで固まるポイント

* 用語（案件/スポンサー/請求/入金）の定義が揃う
* **ステータスとボール（担当/次アクション）が揃う**
* 例外（差し戻し/取り消し/金額変更）の扱いが決まる

**案件登録 → 次アクション設定 → ステータス更新 → 請求作成 → 入金確認**

## Goals：ユーザー像（少人数チームのPM/事務）

**人物像**：2〜5人のチームで兼務しながら複数案件を回している

* 行動：Slackで調整、スプシで一覧、月末に棚卸し
* つまずき：更新漏れ / 形式ゆれ / 重複 / 担当不明 / 請求漏れ
* 理想：誰でも「いま/次/未完了」が分かり、例外でもすぐ復旧できる

---

## プロダクト設計：判断基準（迷わない / 漏れない / 復旧できる）

* **迷わない**：画面を開いた瞬間に「次アクション・担当・期限」が分かる
* **漏れない**：条件付き必須・一意制約・入力制約で“事故”を減らす
* **復旧できる**：履歴と例外導線で「何が起きたか→次にすること」が分かる

> この判断基準に合うなら採用／合わないなら削る

---

## PHASE2：UXの核（一覧→詳細→状態更新）

* 一覧で「次アクション」「期限」「請求予定」が一目で分かる
* 詳細が“唯一の更新場所”（リンク/履歴/メモ/添付）
* 状態遷移でボールの所在が明確になる

## 品質と公開の仕組み（安心の土台）

* **CI（GitHub Actions）**：最低限の品質ゲート（lint / typecheck / test）
* **Preview → 本番**：小さく出してレビュー（合意形成が早い）
* **ログ/切り分け**：原因に辿れる（復旧が速い）
* **回復設計**：入力ミス/重複/例外でも“次にすること”を提示

## 評価計画（公開後に検証する仮説）

※現時点で成果数値は持たない。公開後に測って改善の優先順位を決める。

* **往復が減るか**：確認メッセージ数（週）
* **漏れが減るか**：請求漏れ・更新漏れの発生率
* **棚卸しが短くなるか**：月末棚卸しの所要時間

---

## この設計アプローチが効く領域（参考）

* **アイデア段階のMVP**：要件整理→UX→プロト→公開まで
* **小さな業務アプリ/管理画面**：スプシ/Notion運用を“画面とルール”に置換
* **探索・予約・受付などの体験設計**：迷わず完了できるフロー設計＋実装

**共通する締め**：まず「主要フローが一周できるプロトタイプ」から始め、スプリントで磨いて公開まで持っていきます。
